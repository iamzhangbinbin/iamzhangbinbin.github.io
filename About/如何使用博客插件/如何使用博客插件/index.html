    如何使用博客插件¶

    博客基础¶

    博客是与受众互动的好方法。软件开发人员可以使用博客来宣布新功能、演示其用法并提供背景信息。您可以通过评论最新技术来展示能力，或者将自己的工作记录为最佳实践。关于当前主题的帖子有助于吸引您的主要网站的访问者，并可以保持您的受众的参与度。当然，您可以在博客上讨论您心中的任何主题。
    博客插件使博客与其他内容一起运行变得容易，但如果帖子是您唯一需要的内容，您也可以将其配置为运行独立博客。 在简要概述了博客的基本概念之后，本教程将指导您完成配置博客插件、设置博客、创建文章和定义文章元数据的过程。

    关键概念¶

    帖子、摘录：博客由许多自包含的帖子（通常称为文章）和一个索引页面组成，该页面按时间倒序显示帖子，最近的帖子位于顶部。索引页通常只显示一个简短的摘录和一个链接，用户可以单击该链接以导航到完整的帖子。

    元数据：索引页面和帖子本身都列出了信息，例如您何时发布帖子、何时更新帖子、作者是谁以及预期阅读时间。

    Slug：由于博客文章主要按时间排列，而不是按层次结构排列，因此它们的 URL 并不反映这样的结构。相反，每个帖子的 URL 都包含一个简短的描述，即 slug，它通常派生自帖子的第一个标题。

    导航：主要的导航结构是时间轴，您可以将其细分为多个类别。主索引页面显示较新的文章，而存档部分允许访问按年份组织的较早文章。此外，还可以标记帖子，并且标记索引页面提供基于内容的其他导航结构。

    设置您的博客¶

    博客插件是 Material for MkDocs 的一部分，但您需要在 mkdocs.yml 中对其进行配置。
    如果您尚未这样做，请为您的博客创建一个项目，然后编辑 mkdocs.yml 文件以确保它包含以下内容：

    site_name: Blog Tutorial
site_description: an example blog set up following the tutorial
site_url: http://www.example.com

theme:
  name: material

plugins:
  - search
  - blog
    如果 blog 插件不存在，它将为您的博客文章创建一个目录结构，因此只需运行 mkdocs serve 即可获取： 
    docs
├── blog
│   ├── index.md
│   └── posts
└── index.md
    现在在 docs/blog/posts 中创建您的第一篇博客文章。你可以为你的文章使用任何你喜欢的命名约定和目录结构，只要它们位于 docs/blog/posts 中。

    每个文章都必须有一个页眉，该页眉显示在 Markdown 代码顶部的行之间，带有三个破折号。在此标头中，您至少需要有一个日期条目，但您可以添加其他数据，如下所示。标题之后是页面内容。请注意，有一个 1 级标题很重要，因为插件会使用它来生成 slug。此外，通过向页面添加<!-- more -->，您可以定义索引页面显示的摘录结束位置。

    创建一个文件 docs/blog/posts/myfirst.md，其中包含以下内容： 

    ---
date:
  created: 2023-12-31
---

# Happy new years eve!

We hope you are all having fun and wish you all the best for the new year!
<!-- more -->

Lorem ipsum dolor sit amet, consectetur adipiscing elit, sed do eiusmod
tempor incididunt ut labore et dolore magna aliqua.
    然后，运行 mkdocs serve 并将 Web 浏览器指向 http://localhost:8000/blog。

    blog 插件会自动为博客创建导航元素。索引页仅显示数据提取。当您选择 “Continue reading” 链接时，您将访问完整的博客文章。请注意它是如何从第一级标题生成的 URL。

    帖子元数据¶

    除了日期之外，您还可以提供其他元数据并给插件提供说明，例如将帖子视为草稿或固定它。

    草稿¶

    您可能希望生成博客文章的草稿并在本地使用它，但将其从您发布的内部版本中排除。只需在页眉中添加一个字段，即可表明文章仍处于草稿形式。

    在 docs/blogs/posts/draft.md 中创建第二篇博文，内容如下： 

    ---
date:
  created: 2024-01-01
draft: true
---

# Happy new year!

Happy 2024 to everyone. Wishing you all the best!
<!-- more -->

Lorem ipsum dolor sit amet, consectetur adipiscing elit, sed do eiusmod
tempor incididunt ut labore et dolore magna aliqua.
    现在，请注意草稿在索引页上的显示方式，但带有一个标签，指示它是草稿。当你运行 mkdocs build 时，草稿不会出现在输出中： 
    $ mkdocs build
$ ls site/blog
site/blog
├── 2023
│   └── 12
│       └── 31
│           └── happy-new-years-eve
│               └── index.html
...

    编辑¶

    有时，博主需要更新帖子。当您犯了错误或需要在帖子中反映的内容发生变化时，可能会发生这种情况。要表明您已编辑文章，您可以在页眉中包含更新日期。 更改您的第一篇博文，然后向标题添加编辑日期： 

    ---
date:
  created: 2023-12-31
  updated: 2024-01-02
---
    博客文章本身的 Metadata 部分将包含编辑日期，但默认情况下，索引页面会省略此详细信息。

    阅读时间¶

    为了让读者了解阅读文章可能需要多长时间，系统会自动计算阅读时间。如果要覆盖此项，可以在页眉中执行此作，方法是指定您估计读者阅读文章所需的分钟数。 将阅读时间覆盖添加到您的第一篇博客文章中：

    ---
date:
  created: 2023-12-31
  updated: 2024-01-02
readtime: 15
---

    固定¶

    有时，博客作者想要 “固定 ”一个特定的帖子，这样它就会总是出现在索引页面的顶部，无论发布什么。如果您使用的是 Insiders Edition，则可以通过在页眉中添加 pin 属性来实现此目的：
    将 pin 属性添加到您的第一篇博客文章中： 

    ---
date:
  created: 2023-12-31
  updated: 2024-01-02
readtime: 15
pin: true
---

    相关链接¶

    当您的博客是更广泛的网站（例如技术文档）的一部分时，您将需要提供从博客文章到其他内容的链接。你可以做到这一点的一种方法是有一个 related links 部分。如果您在页面标题中提供链接目标，则博客插件可以为您创建一个：

    将以下内容添加到博客文章中：

    ---
date:
  created: 2023-12-31
...
links:
  - index.md
  - blog/index.md
---
    相关链接显示在 Metadata 部分下。

    这里的好处是你不需要提供页面标题。该插件将通过应用与 MkDocs 用于主导航的相同逻辑来推断链接文本。事实上，语法与mkdocs.yml中 nav 部分的语法相同，因此您可以根据需要覆盖标题，甚至可以定义子部分：

    更改 link 部分以覆盖页面标题： 

    ---
date:
  created: 2023-12-31
...
links:
  - Homepage: index.md
  - Blog index: blog/index.md
  - External links:
    - Material documentation: https://squidfunk.github.io/mkdocs-material
---
    该插件在足够宽的屏幕上在左侧边栏中呈现相关链接，在较窄的屏幕上呈现在帖子底部。更改浏览器窗口的大小以查看实际效果。

    Meta 插件¶

    Meta 插件有助于简化对同一子目录中一组文件通用的元数据的管理。您不必在多个文件的页面标题中重复相同的元数据，而是可以在目录中添加一个 .meta.yml 文件，Meta 插件会将其内容合并到包含的所有页面的标题中。页眉中的设置优先，因此您始终可以通过将设置添加到文章的页眉来覆盖这些设置。
    例如，您可能希望通过将草稿放在一个目录中来管理草稿，这样它们不仅被标记为草稿，而且更容易找到。（否则，您需要检查页面标题或从输出追溯到文件，以找出哪些帖子是草稿。)

    您首先需要在 mkdocs.yaml 中激活插件： 

    plugins:
  - search
  - blog
  - meta
    现在为 draft 创建文件夹：
    $ mkdir docs\blog\posts\drafts
    现在，在此文件夹中，创建一个包含以下内容的文件.meta.yml： 
    draft: true
    添加另一篇博客文章并将其存储在 docs/blog/posts/drafts 中。当您在本地查看它时，您将看到将其标识为草稿的标签，而在为发布而构建的版本中，它不会显示。要将文章从 draft 状态移动到 published，只需将其移动到 drafts/ 之外即可。

    导航、作者和分页¶

    Blog 插件提供博客样式的导航，默认情况下具有按时间倒序排列的索引页面和按年份组织的存档。本教程介绍如何使用类别和 Tags 插件配置默认导航的详细信息、配置作者以及添加更多导航选项。

    集成导航¶

    到目前为止，您已经让 Blog 插件和 MkDocs 来处理导航。对于某些用例，这可能就足够了，而且不在 mkdocs.yml 中声明 nav 部分就足够了。
    但是，您可能希望将博客与在配置的 nav 部分中定义的其他内容和导航结构集成。在这种情况下，您需要提供一个位置，以便 Blog 插件将博客导航附加到导航结构的其余部分。

    将以下内容添加到您的mkdocs.yml，以查看 Blog 插件如何将博客导航与整体导航结构集成。请注意，此时您唯一需要指定的是博客的索引页面，其路径必须与 blog_dir 设置匹配，默认情况下为 blog：

    nav:
  - Home: index.md
  - Install: install.md
  - Usage: usage.md
  - Blog:
     - blog/index.md
    您会注意到 “Blog” 在导航结构中是重复的。为避免这种情况，您可以使用 navigation.indexes 功能将博客索引设置为博客的章节索引页面： 
    theme:
  name: material
  features:
    - navigation.indexes

    配置存档¶

    默认情况下，博客存档仅按年份列出文章。如果要按月添加商品信息，可以配置存档的日期格式。
    将以下内容添加到您的mkdocs.yml以获取带有月份名称的列表（使用在模板选项中选择的语言）： 

    - blog:
    archive_date_format: MMMM yyyy
    例如，如果您不需要完整的月份名称，则可以将日期配置为 MM/yyyy。

    如果要添加日期，您可以为它们添加占位符。例如，要获得美式输出，请将其设为 MM/dd/yyyy。为了让插件按完整日期对博客文章进行排序，您还需要将 archive_url_date_format 设置为包含月份和日期，因此也将其设为 MM/dd/yyyy。

    使用类别¶

    类别是一种使博客文章按主题可访问的方法，同时在每个类别列表中保留基于时间顺序的导航结构。当有一组有限的非重叠类别可以将文章分类时，请使用它们。

    类别显示在主导航中，因此可以从主导航中直接访问。这意味着类别相对较少，否则主导航中的类别部分将变得过于拥挤。

    通过将类别添加到页面标题，将类别添加到您的第一篇博客文章中： 

    ---
date: 2023-12-31
updated: 2024-01-02
categories:
  - Holidays
---
    现在，博客文章已分类，Holidays 显示在主导航中的 Categories （类别） 下，博客文章显示在此类别的索引页中。

    Material 允许您控制博客作者可以使用的类别。您可以在 mkdocs.yml 中声明它们。通过这种方式，您可以确保每个人都遵守商定的类别，并且插件会检测到拼写错误。 在 Blog 插件的配置中添加一个 categories_allowed 条目，其中包含条目 “Holidays” 和 “News”： 

    plugins:
  - search
  - blog:
      archive_date_format: MMMM yyyy
      categories_allowed:
        - Holidays
        - News
    现在，当您向博客文章添加的类别与这两个类别中的任何一个不匹配时，您应该会收到构建错误。

    使用标签¶

    Tags 插件提供了另一种方法，用于对博客文章进行分类，并使其独立于主导航结构进行访问。标记可用于使相关内容易于发现，即使它位于导航层次结构的不同部分。
    您可能有类似的教程以及更全面的设置指南和参考文档。如您所见，可以从标记的页面导航到标记索引，然后从那里导航到带有相同标记的其他页面。

    首先，您需要将插件添加到您的 mkdocs.yml： 

    plugins:
  - search
  - blog:
      archive_date_format: MMMM yyyy
      categories_allowed:
        - Holidays
        - News
  - tags
    完成此作后，您可以为页面标题中的帖子添加标签： 
    ---
date:
  created: 2023-12-31
  updated: 2024-01-02
authors:
  - material
categories:
  - Holidays
tags:
  - new year
  - hogmanay
  - festive season
---
    您应该会在文章顶部看到您定义的标签。然而，目前就是这样。虽然 blog 插件会自动为类别创建索引页面，但 tags 插件不会对标签执行相同的动作。这是因为 tags 插件并非特定于博客。你可以将它用于任何网站内容，因此标签索引应该去掉并不明显。

    您可以使用 Material for MkDocs 公共版本配置基本标签索引。当然，Insider Edition 也支持这一点，但也提供了一种替代索引机制，允许任意数量的标签索引、范围列表、影子标签、嵌套标签等等。

    要使用公有版本配置标签索引，请将 tags_file 条目添加到您的 tags 插件配置中，并在 nav 部分中对其进行配置。请记住在现有 tags 条目的末尾添加一个冒号。 

    plugins:
    - search
    - blog:
        archive_date_format: MMMM yyyy
        categories_allowed:
            - Holidays
            - News
    - tags:
        tags_file: blog/tags.md

nav:
    - Home: index.md
    - Install: install.md
    - Usage: usage.md
    - Blog:
        - blog/index.md
        - Tags: blog/tags.md
    标签索引将附加到配置的页面，您现在应在指定的位置创建该页面。

    请注意，您可以将标签索引页面放在主导航中的任何位置，因此，如果您在其他位置使用标签，而不仅仅是在博客中使用标签，那么您可能希望将标签索引放在导航的博客部分之外。

    定义作者¶

    如果您的博客有多个作者，那么您可能需要确定每篇博客文章的作者。博客插件允许您创建包含作者信息的文件，然后在页面标题中引用特定文章的作者。

    创建包含以下内容的文件 docs/blog/.authors.yml： 

    authors:
  team:
    name: Team
    description: Creator
    avatar: https://simpleicons.org/icons/materialformkdocs.svg
  squidfunk:
    name: Martin Donath
    description: Creator
    avatar: https://github.com/squidfunk.png
    然后在第一篇文章的标题中添加一行： 
    ---
date:
  created: 2023-12-31
  updated: 2024-01-02
authors:
  - team
---
    请注意，authors 是一个列表，因此您可以指定多个作者。

    页码管理¶

    一旦您的博客开始增长，您可能不想关注每页显示的帖子数量。默认情况下，该插件在索引页面上最多显示 10 篇文章。您可以分别更改主索引、档案索引页面和类别索引页面的此编号。

    再添加 5 篇博客文章，然后将分页设置设置为每页仅显示 5 篇： 

    - blog:
    archive_date_format: MMMM yyyy
    categories_allowed:
        - Holidays
        - News
    authors_profiles: true
    pagination_per_page: 5
    您将看到存档和类别页面的分页设置继承自您添加的设置。如果要为不同的索引页设置不同的设置，可以单独指定每个设置： 
    - blog:
    archive_date_format: MMMM yyyy
    categories_allowed:
        - Holidays
        - News
    authors_profiles: true
    pagination_per_page: 5
    archive_pagination_per_page: 10
    categories_pagination_per_page: 10

    博客目录¶

    一旦你有足够多的文章，你可能想做的另一件事是打开为博客索引页面生成目录的功能，让你的读者有机会快速扫描每个页面的内容，寻找他们感兴趣的内容，而无需滚动（假设每页的文章数量不是太大）。

    要为博客索引页面生成目录，请将以下内容添加到博客插件的配置中： 

    - blog:
    blog_toc: true
    archive_date_format: MMMM yyyy
    # ...

    定制 slug¶

    如果出于某种原因，您对 Material for MkDocs 将标题转换为 slug 的方式不满意，您可以创建自己的 slugify 功能，也可以为特定帖子手动定义 slug。

    要定义自己的 slugify 函数，您需要编写一个 Python 函数，该函数在给定配置中的其他参数的情况下将文本转换为 slug。您还需要编写一个返回该函数的函数。

    假设你想定义两个可以在它们之间切换的 slugify 函数。第一个选项返回一个 slug，类似于默认的 slugify 函数生成的 slug。第二个选项将其结果切成单词，并基于其中的最多 5 个返回 slug：

    import re, functools, unicodedata

RE_HTML_TAGS = re.compile(r'</?[^>]*>', re.UNICODE)
RE_INVALID_SLUG_CHAR = re.compile(r'[^\w\- ]', re.UNICODE)
RE_WHITESPACE = re.compile(r'\s', re.UNICODE)

def _make_slug(text, sep, **kwargs):
    slug = unicodedata.normalize('NFC', text)
    slug = RE_HTML_TAGS.sub('', slug)
    slug = RE_INVALID_SLUG_CHAR.sub('', slug)
    slug = slug.strip().lower()
    slug = RE_WHITESPACE.sub(sep, slug)
    return slug

def _make_slug_short(text, sep, **kwargs):
    words = _make_slug(text, sep, **kwargs).split(sep)
    return sep.join(words[:5])

def slugify(**kwargs):
    if 'short' in kwargs and kwargs['short']:
        return functools.partial(_make_slug_short, **kwargs)
    return functools.partial(_make_slug, **kwargs)
    将此代码保存在 ext/slugs.py 中，并添加一个（空的）init.py文件以指示该目录是一个模块。
    现在，您可以像这样配置自定义 slugify 代码： 
    plugins:
- blog:
    # other entries omitted
    post_slugify: !!python/object/apply:ext.slugs.slugify
      kwds:
        short: true
    将博客文章的标题更改为超过 5 个单词，并观察 slugify 函数如何缩短 URL。将 short 属性更改为 false，您可以再次将其关闭。

    如果您只想影响单个博客文章的 slug，则可以通过在文章的标题中指定它来手动定义它。请注意，这是最后的选择。为每个帖子手动指定自定义 slug 会很乏味。
    例如，如果您希望 slug 是 'ny-eve' 而不是有点长的 'happy-new-years-eve'，您可以添加以下内容： 

    ---
date:
  created: 2023-12-31
  updated: 2024-01-02
readtime: 15
pin: true
slug: ny-eve
---
    此帖子的 URL 现在应为 http://localhost:8000/blog/2023/01/31/ny-eve/。

    参与和传播¶

    通过提供人们可以订阅的 RSS 源和集成讨论系统，您可以促进读者参与并改进博客上内容的传播。要详细了解谁在阅读您的帖子或谁没有阅读您的帖子，您可能需要集成一个分析系统。当您发布新的博客文章时，您可能还想在社交媒体上发帖。本教程为您介绍了所有这些主题。

    RSS 源¶

    RSS 源允许用户订阅博客，以便在您发布新文章时收到通知。RSS 源阅读器通常用于访问用户关注的博客。他们通常支持下载博客内容以供离线使用。

    为您的博客创建 RSS 提要的一种简单方法是使用 MkDocs RSS 插件，它与 Material for MkDocs 很好地集成在一起。由于它是第三方插件，因此您需要在使用前安装它。

    将 RSS 插件安装到您的项目中： 

    $ pip install mkdocs-rss-plugin

    请务必按照基本博客教程中的说明配置 site_name、site_description 和 site_url 设置。RSS 插件使用此信息来构建源，因此请确保您已配置它们。

    现在，在 mkdocs.yml 中配置插件。提供的选项将为其创建 RSS 条目的页面限制为博客文章，这可能就是您想要的。另请注意日期字段的配置，以匹配 Material for MkDocs 用于容纳创建日期和更新日期的格式。 

    plugins:
    - ...
    - rss:
        match_path: "blog/posts/.*"
        date_from_meta:
          as_creation: date.created
          as_update: date.updated
    请查看 http://localhost:8000/feed_rss_created.xml 以查看 RSS 源的所有 XML 荣耀。您可以使用可以显示原始 RSS 源的 Firefox 或 Chrome 等浏览器，也可以使用 curl 获取源并使用 xmllint 对其进行格式化。（您可能需要安装这些工具。） 
    curl -s http://localhost:8000/feed_rss_created.xml | xmllint --format -
    您可能还想尝试使用 Feed 阅读器来验证 Feed。有各种桌面和移动应用程序以及在线服务。当然，要使用后者，您需要将项目部署到他们可以访问的地方。

    社交媒体按钮¶

    社交媒体按钮有两个目的：允许您的读者导航到您的社交媒体资料或分享您通过他们自己的帐户发布的内容。

    个人资料链接¶

    指向社交媒体资料的链接通常在页面页脚, Material for MkDocs使这变得容易。您需要做的就是提供必要的链接并定义要使用的图标。

    向 mkdocs.yml 添加一个 extra 部分，并在其中添加一个社交部分以包含链接定义列表。这些包括要使用的徽标和指向配置文件的链接。 

    extra:
  social:
    - icon: fontawesome/brands/mastodon
      name: squidfunk on Mastodon
      link: https://fosstodon.org/@squidfunk
    对于图标，您可以选择与主题捆绑在一起的图标的任何有效路径。该名称将用作图标的 title 属性，包含此属性可提高辅助功能。对于流行的社交媒体系统，链接必须是绝对的，并且需要包含方案，很可能是 https://。

    您也可以使用其他方案。例如，要创建允许用户创建电子邮件的图标，请添加以下内容： 

    extra:
  social:
  - icon: /fontawesome/regular/envelope
    name: send me an email
    link: mailto:<email-address>

    最后，您可以在站点中指定一个 URL，例如指向您的联系页面的 URL。可以仅指定页面的路径： 

    extra:
  social:
  - icon: /material/mailbox
    name: contact us
    link: /contact

    “共享和点赞”按钮¶

    添加让人们在社交媒体上分享您的内容的按钮要复杂一些，这就是为什么有些公司为此提供组件的原因。

    这种共享按钮的实现特意不使用第三方代码。它支持共享到 Twitter/X 和 Facebook，而不会在有人查看页面时导致数据流向这些公司。只有当有人点击分享按钮时，才会与这些公司的服务器进行交互。

    为了添加共享按钮，您可以添加一个 hook，该 hook 附加用于共享当前页面的按钮。 在项目根目录中创建一个目录钩子，并在 mkdocs.yml 中配置它： 

    hooks:
  - hooks/socialmedia.py
    使用以下 Python 代码添加文件 hooks/socialmedia.py： 
    from textwrap import dedent
import urllib.parse
import re

x_intent = "https://x.com/intent/tweet"
fb_sharer = "https://www.facebook.com/sharer/sharer.php"
include = re.compile(r"blog/[1-9].*")

def on_page_markdown(markdown, **kwargs):
    page = kwargs['page']
    config = kwargs['config']
    if not include.match(page.url):
        return markdown

    page_url = config.site_url+page.url
    page_title = urllib.parse.quote(page.title+'\n')

    return markdown + dedent(f"""
    [Share on :simple-x:]({x_intent}?text={page_title}&url={page_url}){{ .md-button }}
    [Share on :simple-facebook:]({fb_sharer}?u={page_url}){{ .md-button }}
    """)
    hooks 首先检查当前页面是否是博客文章，然后为分享按钮附加 Markdown 代码。这些按钮使用图标，因此您还需要配置以下 Markdown 扩展： 
    markdown_extensions:
  - attr_list
  - pymdownx.emoji:
      emoji_index: !!python/name:material.extensions.emoji.twemoji
      emoji_generator: !!python/name:material.extensions.emoji.to_svg

    添加讨论系统¶

    允许您的读者对您的帖子发表评论是接收反馈、学习一些东西以及让读者有机会讨论内容及其主题的好方法。
    那里有很多讨论系统，在选择适合您的博客的系统时，您需要考虑您的受众。同样，您还需要考虑对沟通渠道的现有承诺。例如，如果您是 Slack 的重度用户，则此系统可能具有字符串首选项。请注意，当您添加通信渠道时，您需要准备好定期使用它并主持讨论。

    Giscus 集成¶

    在本教程中，我们将使用 Giscus，因为它是免费的、开源的，并使用 GitHub Discussions 作为后端。因为 Material for MkDocs 的许多用户都使用 GitHub，所以这似乎是一个显而易见的选择。

    要将 Giscuss 添加到您的博客中，您需要完成许多步骤：

    1. 如果还没有 GitHub 存储库，请创建一个 GitHub 存储库
    2. 打开讨论并安装 Giscus 应用程序
    3. 配置将 Giscus 嵌入到您的博客中所需的代码
    4. 将代码添加到您的 MkDocs 项目

    您可能希望为本教程创建一个测试存储库，以便稍后将其废弃。以下说明假定您是用户 “example” 并且您创建了一个存储库 “giscus-test”。存储库需要是公开的，以便人们能够使用讨论。

    在下面给出的说明中，如果您选择其他名称（例如，当您想直接处理现有存储库时），您至少需要替换用户名和存储库名称。

    设置仓库后，转到其设置页面并在 General （常规） 部分找到 Features（功能）。勾选 Discussions 复选框。您将看到 Discussions （讨论） 显示在仓库的顶部导航中。如果您使用的是实时仓库，那么此时您可能希望向 dicussions 部分添加一些最少的内容，然后返回本教程。

    接下来，您需要按照这句话中的链接安装 Giscus 应用程序，选择 Install（安装），然后按照说明选择要安装 Giscus 应用程序的位置：

    1. 选择要使用的仓库的账户或组织。
    2. 选择 Install only on select repositories （仅在选定的存储库上安装），然后选择要使用的存储库。请注意，您可以在此处选择多个存储库。
    3. 选择最后的 Install。您可能需要进行身份验证才能授予权限。
    4. 您最终将进入设置中的应用程序页面，您可以在其中控制 Gicsus 应用程序并在需要时卸载它。

    这就是仓库所需的全部准备工作。接下来，是时候生成一段代码，将 Giscuss 嵌入到你的网站中了。生成的代码片段将如下所示： 

    <script src="https://giscus.app/client.js"
        data-repo="<username>/<repository>"
        data-repo-id="..."
        data-category="Announcements"
        data-category-id="..."
        data-mapping="title"
        data-strict="1"
        data-reactions-enabled="1"
        data-emit-metadata="1"
        data-input-position="top"
        data-theme="preferred_color_scheme"
        data-lang="en"
        data-loading="lazy"
        crossorigin="anonymous"
        async>
</script>

    转到 Giscus 主页并配置嵌入代码。有许多设置：

    1. 选择语言
    2. 输入用户名/组织名称和存储库名称
    3. 选择如何将讨论映射到您博客上的页面。由于对于博客文章，标题是 URL 的基础，因此使用 Discussion title contains page <title> 选项是有意义的。
    4. 在 Discussion Category （讨论类别） 下，选择 Announcements （公告） 以将新讨论的创建限制为 Giscuss 和具有维护员或管理员权限的人员。
    5. 在 Features （功能） 下，选择以下选项：
      a. 为主帖子启用回应
      b. 发出讨论元数据
      c. 将评论框放在评论上方
    6. 在 Theme （主题） 下，选择 Preferred color scheme （首选配色方案），以便 Giscus 与用户为您的网站选择的颜色方案匹配。

    完成这些设置后，您现在需要将代码集成到您的网站中。为此，存在一个部分 partials/comments.html，并且默认为空。它包含在 content.html 部分，因此将包含在您网站上的每个页面中。你可能想要也可能不想要这个。在本教程中，您将 Giscus 集成限制为仅博客文章，但如果您想让每个页面的 Giscus 讨论处于活动状态，则很容易实现此目的代码。

    首先，您需要创建一个 overrides 目录，该目录将包含要覆盖的模板和部件。 

    mkdir -p overrides/partials
    你需要在 mkdocs.yaml 中声明它： 
    theme:
  name: material
  custom_dir: overrides
    现在添加一个文件 overrides/partials/comments.html 并粘贴您从 Giscus 主页获得的代码片段。在本地查看结果，您将看到集成在网站的所有页面上都处于活动状态。如果你想把它限制在你的博客文章中，你需要在 Giscus 脚本周围添加一个条件，用于测试是否应该包含评论。执行此作的一种简单方法是测试元数据标志： 
    {% if page.meta.comments %}
<script>...</script>
{% endif %}
    缺点是你现在需要手动打开每篇博客文章的评论——除非你想关闭某些博客文章的评论。要获取所有博客文章的评论部分，请使用如下代码： 
    {% if page.file.src_uri.startswith('blog/posts') %}
<script>...</script>
{% endif %}
    现在您应该看到 Giscus 评论已添加到博客文章的底部，但未添加到其他页面上。
